Ubuntu

Ubuntu Linux ist eine Linux distribution auf Debian basierend.

SQLite

  • ist eine kommandozeilenbasierende SQL-Datenbank für Linux / Max / Windows
  • Es ist sicher, schnell und “reduced to the max”

  • eine Datenbank kann mehrere Tabellen haben (z.B. customers, movies, orders)
  • eine Tabelle kann mehrere Attribute haben (im Excel waren es Spalten, z.B movie od. email)
  • Unterschied ist im SQLite hat jedes Attribut ein Datentyp (z.B. “movie” TEXT NOT NULL)
  • Es gibt nur INT und STRING als Datentypen
  • Es gibt auch Primärschlüssel Attribut, welches als ID verwendet wird
  • Die einträge innerhalb einer Tabelle sind Datensätze

DB Browser for SQLite

  • mit dem DB-Browser hat man eine visuelle Darstellung (UI) des SQL-files
  • man kann darin die Tabellen, Datenbanken und auch Daten visuell sehen (Daten als sheet, Tabellen in einer Hierarchie-Struktur)
  • Man kann damit Daten löschen / Hinzufügen / filtern / sortieren
  • Im Tab “SQL ausführen” kann man dann manuell mit der SQL-Sprache Queries ausführen
    • Kann zum Beispiel sein 
    SELECT * FROM customers WHERE postal_code > 8000
    • Aber kann eigentlich alles, also auch bearbeiten, löschen und Beziehungen zwischen Tabellen erstellen

RDBMS

  • RDBMS heisst Relationales DatenBankManagementSystem
  • Das system legt somit auch Beziehungen zwischen Daten fest
  • Eine relationale Datenbank besteht also aus einer Sammlung von Tabellen

Beziehungen

  • Es gibt eine Primärtabelle mit Primärschlüsseln
    • auf diese wird später verwiesen
  • Und eine Detailtablle mit einem Fremdschlüssel
    • diese verweist auf die Primärtabelle
    • wenn dies nicht klappt ist das meistens wegen “Inkonsistenz”
  • Sie sind vom gleichen Datentyp
  • Und jeder Fremdschlüsselwert muss in der Primärtabelle mit einem Primärschlüssel übereinstimmen

Zum Beispiel

1:m

  • Primärtabelle “Customers”
  • Detailtabelle “Orders”
  • Jeder order hat einen Customer
  • Aber ein Customer kann mehrere Orders haben
  • es ist also eine 1:m Beziehung, die Primärschlüssel können mehrere male in der Detailtabellen verlinkt werden, aber nicht anders rum

m:m

  • Zum Beispiel Orders & Movies
  • Ein Order kann 1 oder mehrere Movies haben
  • Gleichzeitig kann 1 Movie 1 oder mehrere Orders haben
  • Wenn man nun eine Order-Tabelle hat, hat man das Problem, dass wenn in einer Bestellung mehrere Filme gekauft wurden, die Bestellung in mehrere Zeilen aufgeteilt werden muss, eine pro Film
  • Man kann dies verhindern, indem man eine Zwischentabelle macht, welche die Movies zu den Bestellnummers verlinkt
  • Nun braucht man pro Bestellung nur eine Zeile

Beziehungstypen

  • 1
    • Kann mit nur einer anderen Entität verbunden werden
  • c
    • Konditionelle 1, kann auch 0 verbindungen haben
  • m
    • Kann mit einer oder mehreren Entitäten verbunden werden
  • mc
    • Konditionelles m, kann auch 0 Verbindungen haben

  • In relationalen Datenbanken sind nur 1:c, 1:mc, & mc : mc möglich
    • Die Beziehungen sind also immer konditionell

Ergänzungen zum Primärschlüssel

  • Die Vergabe ist beliebig, da die Reihenfolge in einer Tabelle keinene Aussagekraft hat
    • Es kann also vorkommen, dass gewisse Werte nicht existieren, z.B. 1, 2, 3, 6, 9
  • Der Wert muss jedoch eindeutig sein
  • Es müssen ganze, positive Zahlen sein
  • Jede Tabelle sollte einen Primärschlüssel definieren
  • Das DBMS macht das normalerweise automatisch (ATUINCREMENT)

Beispiele komplexere Queries

SELECT last_name AS Nachname, first_name AS Vorname, SUM(amount_chf) AS Betrag
FROM customers AS c
JOIN orders AS o
ON o.customer_id = c.id
WHERE last_name = "Watman"
AND first_name = "Omar";

→ Den Totalbetrag von allen Filmkäufen des Kunden Omar Watman anzeigen als Tabelle
