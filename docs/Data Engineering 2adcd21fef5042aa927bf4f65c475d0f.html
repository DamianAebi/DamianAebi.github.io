Data Engineering

Datenbanken sind feste Speicher für Informationen. Diese Informationen sind strukturierte Daten.

Sie werden normalerweise von einem DBMS (z.B. MySQL oder Postgress) verwaltet

Beispiel Datenbank “user”

IDnamevornamegeburtsdatum
1tomevmarjan…
2röthlisbergerben…
  • Zeile = Datensatz / Entität / Tupel
  • Spalte = Attribut
  • Entitätsmenge = “user”, die ganze Tabelle
    • Achtung: Name der Entitätsmenge sollte stets im Singular geschrieben sein
  • Datentyp = Domäne
  • Zelle = Attributwert

Semistrukturierte Daten

Daten wie z.B. Word-Files

ORM (Object Relation Mapping)

  • Sind ein Tool, welches gewisse Objekte / Ausschnitte einer Datenbank
  • Man muss nicht mehr ein SQL query brauchen damit
  • Kann SQL-Queries in Objekte umwandeln und umgekehrt, damit der Entwickler kein SQL brauchen muss

NoSQL

  • Nicht relational im vergleich zu SQL
  • Kann schneller sein

Beziehungen (Kardinalitäten)

  • m:m brauchen immer eine Zwischentabelle
  • ERD = logisches Datenmodell
  • Vereinfachtes ERD = ERD, welches wir bei der Gibb gelernt haben (selbes aber keine Zwischentabellen und Fremdschlüssel)
  • ERM = konzepitonnelles Datenmodell
    • Entität = Rechteck, z.B. Song
    • Attribute = Kreis (z.B. Titel)
    • Beziehung = Raute (z.B. “hat”)
    • Raute ist zwischen zwei Entitäten (z.B. zwischen Album und Song) verbunden mit einem Strich, wo wiederum die Beziehungen aufzählt
  • Echte Datenbank = physisches Datenmodell

Datentypen

  • char
    • beliebige Anzahl Zeichen (nicht wie in C# ein char)
    • braucht immer die Menge an Speicher, welche angegeben wurde
  • varchar
    • beliebige Anzahl Zeichen
    • jedoch verwendet er nur den nötigen Speicher + ein Byte um festzuhalten, wieviel er bereits gepseichert hat
  • blob
    • Sammlung an Werten
    • können auch unstrukturierte Daten sein
    • braucht man jedoch praktisch gar nicht mehr, weil auch z.B. Bilder heutzutage auf einem separaten storage gespeichert werden und nur der Link zum Bild ist in der Datenbank
  • boolean
  • integer
  • float
  • datetime
INT / INTEGER	Ganze Zahlen, +/- zwei Milliarden (genauer von -2147483648 bis 2147483647)
DOUBLE	Zahlen mit Kommastellen, Achtung ((10 / 3) * 3) != 10
DECIMAL	Kommastellenzahlen für höhere Präzision (Beispielsweise Geld oder Devisen)
CHAR	Zahlen und Buchstaben, fixer Speicherbedarf, Länge 0 bis 255 Zeichen
VARCHAR	Zahlen und Buchstaben, variabler Speicherbedarf, Länge 0 bis 65535 Zeichen
TEXT	Zahlen und Buchstaben bei grösseren Texten, analog wie BLOB
TIME	Zeit,	Format:	HH:MI:SS
DATE	Datum,	Format:	YYYY-MM-DD (01.01.1000 – 31.12.9999)
DATETIME	Datum-Zeit, 	Format:	YYYY-MM-DD HH:MI:SS (01.01.1000 – 31.12.9999)
TIMESTAMP	Zeitpunkt, 	Format:	YYYY-MM-DD HH:MI:SS (01.01.1970 – 19.01.2038)
BLOB	Binary Large Object, binäre Daten (EXE-Files, Bilder, Videos etc.)

Wichtigkeit der Planung

Es ist generell wichtig, dass von Anfang an die Attribute die richtigen Datentypen & constraints haben, da es immer aufwändiger ist, etwas im Nachhinein zu ändern, als wenn es von Anfang an richtig ist. Hier sind auch zwei Fehler die häufig gemacht werden:

  • Postleitzzahlen sollten als varchar gespeichert werden
  • Telefonnummern sollten auch als varchar gespeichert werden

Referentielle Integritätsbedingungen

Sind Bedingungen, die definiert werden, um die Integrität sicherzustellen.

  • z.B., was passiert, wenn eine Entität gelöscht wird, auf welches ein Fremdschlüssel verwies
  • Cascade = Alles, was referenziert hat, wird gelöscht oder überarbeitet
  • Set Null = Der Fremdschlüssel wird mit NULL ersetzt
  • Restrict / No Action = Es ist nicht möglich eine Entität zu löschen, solange darauf referenziert wird

Natürlicher Primärschlüssel vs Surrogate key

  • Natürlicher Primärschlüssel
    • z.B. eine AHV Nummer kommt in der Natur vor und ist bereits ein einzigartiger Wert, kann also als Primärschlüssel verwendet werden
    • Sollte niemals verwendet werden, weil es immer ein Risiko gibt, dass dieser Schlüssel sich ändert.
  • Surrogate key
    • Ist alles, was kein natürlicher Primärschlüssel ist, also z.B. 0, 1, 2

Performance Scaling

  • Vertical scaling
    • Verbesserung der Maschine (mehr RAM, mehr CPU, mehr Speicher etc.)
  • Horizontal scaling
    • Neue Maschinen hinzufügen, anstatt eine Maschine zu verbessern

Wenn man dynamisch horizontal skaliert nennt man es “Load-Balancing”

  • Der Load-Balancer bekommt die Anfragen der Nutzer und entscheidet, auf welche Maschine zugegriffen werden muss
  • Die Maschinen hingegen greifen dann wieder auf die selbe Datenbank zu, was wiederum zu Überlastung der Datenbank führen kann
  • Man kann dies optimieren, indem man mehrere synchronisierte Datenbanken hat, welche parallel abgerufen werden können
  • Oft wird um die Synchronisierung zu verhindern ein vertikales Scaling für Datenbanken verwendet
  • Jedoch für die Backend Maschinen braucht man horizontales scaling. Je nach Anzahl aktive user können automatisch und agil mehr Maschinen aufgeschaltet werden

Es brauch nicht unbedingt eine Maschine zwischen User und Server (Monolyt), also Frontend und Backend sind auf derselben Maschine

Jedoch ist die Datensicherheit nicht mehr so gut. Ausserdem kann man danach nicht mehr mehrere Frontends mit demselben Backend hat

Performance

Indexierung

Verschnellern das Abrufen von Werten, indem Attributen einen Index enthalten und dazu ein Binärbaum gebaut wird. Die Abfrage ist nun schneller, jedoch ist das Löschen und verändern der Werte langsamer.

Wenn die Datenbank viele Einträge hat, kann man indexieren.
Wie Register in einer Datenbank mit Pointern zu werten. Damit können schnellere Suchalgorithmen gemacht werden.

Theoretisch macht es kein Sinn, wenn die Daten schon sortiert sind. Es macht auch kein Sinn, wenn man nur wenig Daten hat. 

CREATE INDEX idx_email ON Benutzer (Email);
# Index ist gesetzt. Wenn man nun einen SELECT-Befehl auf "Email" macht,
# ist es, je nach grösse der DB, massiv schneller.

# (Der SELECT Befehl muss nicht den Index ansprechen, der Index wird automatisch verwendet,
# wenn es auf diesem Attribut einen gibt)

Es gibt auch Unique Indexe. Das bedeutet, dass z.B. in der indexierten Vornamen Tabelle nicht zweimal ein selber Vorname vorkommen darf.

CREATE UNIQUE INDEX kundenid_indx ON kunden (KundenID);

Es gibt auch zweispaltige Indexe. Diese sind so sortiert, dass wenn die erste Spalte Duplikate enthält, dass danach nach der zweiten Spalte sortiert. 

CREATE INDEX joined_index ON bestellpositionen (BestellID, ProduktID);

Man kann auch einen Volltext indexieren, um schnell ein Wort im text zu finden.
